我今年的新年决心之一是写更多博客。我从头开始重新构建了博客。这是之前/之后：

这次重建有很多目标，但是主要目的是给我一个我可以更认真地使用的平台。尽管它仍然是个人博客，但我希望它看起来更像是一个社区资源，人们逐渐将其视为优质教程和文章的良好来源。

该目标的很大一部分是定期发布！我设定的目标是每周至少发布 1 条帖子。自发布以来，我一直努力实现这一目标，并希望保持这一趋势。

尽管我的读者已经接受了大多数更改，但是有一个更改使一些人感到惊讶：我的新博客不是开源的。我对此有很多疑问，并认为解决这个问题并分享我的一些推理 😊

降低摩擦

其中很大一部分是围绕摩擦。为了使新帖子的发布速度保持较高水平，我需要发布时尽可能做到轻松自如。拥有一个开源博客会在某些方面增加摩擦。

源代码控制

我的博客使用 MDX，一种 Markdown 的精美版本，可让我嵌入 React 组件。这些帖子会被检入 Git 存储库中，而不是存储在 CMS 中。

每个帖子顶部都有个 isPublished 的布尔值。在任何给定的时刻，我都有 1 至 5 个草稿职位。有时，草稿会持续数周，然后我才能完成并发布它！

如果此博客是开源的，那么所有这些草案都是公开的。我不喜欢这个注意 😂 我想控制帖子何时公开发布！

有解决方案。我可以将帖子保存在单独的仓库中，也可以使用 Git 子模块。但是，这会使我的过程复杂化。而且过程越复杂，我从事该过程的动力就越低。

计划赶不上

由于这些原因，我现在暂时将博客保持闭源状态。

我认识到这对于那些看到很棒的东西并想知道我是如何做到的人来说实在是太可惜了。但是我确实有针对这种情况的计划！

我宁愿为详细的功能编写详细的教程，也不愿开源庞大而混乱的代码库。这已经开始，请查看以下帖子：

  1. 追求完美的黑暗模式，适用于现场的明暗模式逻辑

  2. 使用 CSS 计数器，以用于有序列表（例如这个！）

  3. 神奇的彩虹渐变，用于时事通讯的“订阅”按钮

我喜欢这个策略，因为它是如此明确。使用此博客中的大多数代码，大量的上下文仅存在于我的脑海中。如果您要仔细阅读代码，则必须填写该上下文，这可能会导致问题。这样，我可以指导您了解事物的工作方式，权衡因素，局限性。包括上下文。
